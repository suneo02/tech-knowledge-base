# 文档编写规范

## 快速准则
- 优先用图/表表达，再补充必要说明；代码片段控制在 10 行内。
- 所有文档按模块放在 `docs/`，引用采用相对路径并与代码互相 `@see`。
- 内容仅覆盖完成功能所需的交互、数据与流程，性能/监控/运维等非功能信息不写或移至附录。
- 默认站在前端视角编写，涉及跨端内容需显式标注。
- 每篇文档控制在 1 页（约 120 行）内，聚焦“是什么、为什么、怎么做”。

## 常见文档角色
| 类型 | 目的 | 关键内容 |
| --- | --- | --- |
| README | 目录速览 | 作用一句话、结构树、关联文档 |
| 需求 | 说明要做什么 | 目标、页面/交互、数据、限制 |
| 设计 | 拆解怎么呈现 | 信息架构、页面蓝图、状态与流程 |
| Issue | 记录问题闭环 | 现象、根因、影响、方案、验证 |

更多细节见对应规则：`readme-rule.md`、`require-doc.md`、`design-doc.md`、`issue-doc-rule.md`。

## 编写流程
1. 明确目标受众与阅读时长（一般 ≤5 分钟）。
2. 先绘制 Mermaid 图或列出结构表，再补充文字说明。
3. 用对勾/编号列出关键要求或决策，避免散文式描述。
4. 每条内容核对是否直接支撑功能交付，不相关的非功能信息删除或外链。
5. 在正文中附上代码或设计链接，并让相关代码 `@see` 回指。
6. 更新完成后在 PR 中标注同步的文档路径。

## 检查清单
- [ ] 图表覆盖主要流程或结构，文字仅补充要点。
- [ ] 表格字段齐全，列顺序为：名称、类型、说明/限制、来源。
- [ ] 链接均为相对路径，点击可达。
- [ ] 文档长度控制在 1 页内，无冗余背景或非功能描述。
- [ ] 代码或 PR 中已加入反向引用。
