# 需求文档编写规范

## 快速准则
- 聚焦“要做什么”，不写实现细节或技术选型。
- 以用户视角描述界面、操作、数据、反馈，句式短平快。
- 只写影响功能体验的规则，性能、监控、可运维等非功能内容移出需求文档。
- 全文控制在 1 页内，图/表优先于长段文字。

## 推荐结构
1. **功能概览**：1-2 句说明目标用户与使用场景。
2. **页面/模块**：用示意图或列表说明区域划分与主要元素。
3. **用户流程**：用流程图或编号步骤描述关键任务。
4. **数据与规则**：表格列出展示字段、输入校验、状态切换，只保留与功能相关的约束。
5. **反馈与异常**：列明成功、失败、加载、空态的用户感知。

## 写作提示
- 每个操作都回答“谁执行”“触发条件”“界面反馈”。
- 复用既有组件时引用设计文档或代码路径，而非重新描述交互。
- 差异化需求（如权限、灰度策略）单独列出，便于评审；其余非功能项移至专门文档引用。

## 检查清单
- [ ] 目标与场景明确，阅读后能直接进入设计评审。
- [ ] 页面结构与流程通过图或表呈现。
- [ ] 数据字段写明来源、格式、校验。
- [ ] 每种状态都有反馈描述，且不引入无关的非功能说明。
- [ ] 文内引用均为相对路径并 `@see` 到代码或设计。
