# GEL Workspace Monorepo 面试问题

> 基于 [gel-workspace-monorepo-facts.md](./gel-workspace-monorepo-facts.md) 的第一版项目问答稿。

## 基础题

### 1. 这个 Monorepo 项目最初是为了解决什么问题？

- 分类：基础
- 风险等级：低
- 回答锚点：`gel-workspace-monorepo-facts.md` -> `项目背景`
- 答题提示：
  - 迁移前团队约 4 人，已有主 PC 应用和 2 到 3 个报告/打印应用。
  - 当时没有成型公共包，跨项目逻辑复用较弱。
  - 新 AI 聊天项目启动后，统一管理和复用诉求开始变强。
- 参考回答：
  - 这个 Monorepo 不是为了赶时髦去做的，它最初解决的是复用和协作问题。迁移前我们大概 4 个人，已经有一个主 PC 应用和几个报告、打印类应用，但当时没有成型的公共包，很多逻辑跨项目不好复用。后来新 AI Chat 项目要启动，它在业务上又和原来的系统属于同一组大功能，这时候如果还继续散在多个仓库里，后面公共能力和任务协作都会越来越难维护，所以我才主导把它收进一个统一工作区里逐步演进。

### 2. 你在这个项目里的角色到底是什么？

- 分类：基础
- 风险等级：高
- 回答锚点：`gel-workspace-monorepo-facts.md` -> `我的角色边界`
- 答题提示：
  - 主导迁移方案。
  - 亲手做了仓库结构、包边界、第一批公共包拆分、`turbo` pipeline、新应用构建方式。
  - 不要把全部迁移脚本和所有工程化细节都揽到自己名下。
- 参考回答：
  - 我在这个项目里的角色，更准确说是迁移方案 owner。我主导了整体迁移思路，也亲手做了比较关键的几块，比如仓库结构、应用和包的边界、第一批公共包拆分、`turbo` pipeline，以及新应用的构建方式落地。但我不会把全仓所有工程化细节都揽到自己头上，比如不是所有迁移脚本都由我写，也不是旧主应用所有构建改造都由我一次性重写。我更愿意把自己定义成主导方案并完成关键落地的人，而不是包装成一个人独立做完全部迁移。

## 决策题

### 3. 为什么当时要做 Monorepo，而不是继续多仓库？

- 分类：决策
- 风险等级：中
- 回答锚点：`gel-workspace-monorepo-facts.md` -> `关键决策 / 1. 采用 Monorepo`
- 答题提示：
  - 核心是逻辑复用需求开始明确出现。
  - 新 AI 聊天项目在业务上仍属于同一组大功能。
  - 多仓库下公共逻辑复用路径差、维护成本高。
- 参考回答：
  - 我当时选择 Monorepo，核心原因是复用需求已经从“以后可能需要”变成“现在就需要”。尤其 AI Chat 项目起来以后，它并不是完全独立的新产品，而是和原来几个应用还共享不少能力。如果继续多仓库，公共逻辑要么复制粘贴，要么额外搞一层很重的发布和接入流程，维护成本会越来越高。Monorepo 对当时的团队规模和协作方式更合适，它可以先把工作区统一起来，再逐步把公共能力抽出来，不需要一开始就把所有东西都包管理化到极致。

### 4. 为什么不是一次性迁移，而是渐进式迁移？

- 分类：决策
- 风险等级：中
- 回答锚点：`gel-workspace-monorepo-facts.md` -> `关键决策 / 2. 采用渐进式迁移`
- 答题提示：
  - 新项目先进入 Monorepo。
  - 旧项目在合适且必要的时机再逐步迁入。
  - 主应用先统一工作区与任务编排，再逐步接入公共包复用。
- 参考回答：
  - 我没有做一次性迁移，主要是因为当时最重要的不是“形式上全都进 Monorepo”，而是先把真正有收益的部分做出来。所以我的策略是新项目先直接进入 Monorepo，旧项目则在复用需求足够明确、迁进去确实有意义的时候再逐步纳入。主应用也是一样，先保证它在同一个工作区里，统一依赖和任务编排，公共包复用后面再一点点接。这样做的好处是风险可控，不会因为追求一步到位把已有业务节奏打乱。

### 5. 为什么第一批先拆 `types` 和 `util`，而不是直接拆 `ui`？

- 分类：决策
- 风险等级：中
- 回答锚点：`gel-workspace-monorepo-facts.md` -> `关键决策 / 3. 公共包按依赖层次逐步拆分`
- 答题提示：
  - `types` / `util` 依赖更底层，更容易稳定。
  - 多应用复用最直接，抽离风险最低。
  - `ui` 会牵涉样式、React 版本和更多运行时依赖。
- 参考回答：
  - 第一批先拆 `types` 和 `util`，是因为这两类东西依赖最底层，最容易稳定下来。它们基本是纯 TS 类型或者纯工具逻辑，很多应用都能直接复用，抽出来以后回归风险也比较低。相反如果一开始就拆 `ui`，马上会碰到样式、运行时依赖、React 版本这些问题，复杂度会明显上升。所以我当时是按依赖层次在拆，先拿最底层、最稳、最容易形成公共能力的部分开刀，等基础包稳定了，再逐步往 `api`、`ui` 和更业务化的包扩展。

### 6. 为什么需要 Turborepo，不能只用 `pnpm workspace` 加脚本吗？

- 分类：决策
- 风险等级：中
- 回答锚点：`gel-workspace-monorepo-facts.md` -> `关键决策 / 4. 引入 Turborepo`
- 答题提示：
  - `build` / `tsc` / `dev` 执行顺序主要靠人工维护。
  - 没有缓存只是问题一部分，更大的问题是任务依赖关系难维护。
  - `turbo` 带来依赖驱动执行、按应用过滤和统一根命令入口。
- 参考回答：
  - 只用 `pnpm workspace` 加脚本当然也能跑，但问题是当项目一多，`build`、`tsc`、`dev` 这些任务的执行顺序很快就开始依赖人工维护。没有缓存只是其中一个问题，更麻烦的是改了一个公共包之后，到底哪些任务要重新跑、哪些可以并行、哪些必须串行，脚本会越来越难维护。我引入 `turborepo` 更看重的是任务编排能力，让任务按依赖关系自动执行，也支持按应用过滤和统一从根命令触发。这样开发体验和维护成本都会更稳定，不再主要靠人脑记忆。

## 深挖题

### 7. 你实际是怎么决定哪些逻辑该抽成公共包的？

- 分类：深挖
- 风险等级：高
- 回答锚点：`gel-workspace-monorepo-facts.md` -> `拆包时的真实判断标准`
- 答题提示：
  - 不看“像不像能复用”，而看抽象纯度、依赖复杂度、复用场景和影响面。
  - 强耦合逻辑不直接强拆。
  - 常见处理方式是只抽核心部分，并把外部依赖参数化。
- 参考回答：
  - 我决定要不要抽成公共包，不是看它“像不像能复用”，而是先看它到底纯不纯。很多逻辑表面上多个地方都在用，但里面其实已经耦合了应用内部依赖，这种我不会直接硬拆。更常见的做法是先只抽其中最核心、最稳定的部分，再把原来依赖的外部信息改成参数传入。如果一个逻辑复用场景还不明确，或者影响面太大、回归风险高，我就会先降低优先级。我的判断维度主要是抽象纯度、依赖复杂度、复用需求和回归风险，这样拆出来的包才真的能长期维护。

### 8. 你亲手做的第一批落地动作有哪些？

- 分类：深挖
- 风险等级：中
- 回答锚点：`gel-workspace-monorepo-facts.md` -> `落地动作`
- 答题提示：
  - 建仓库结构。
  - 规划 `apps` / `packages` 边界。
  - 先拆 `types` / `util`，再拆 `api` / `ui`，最后拆业务包。
  - 配 `turbo` pipeline，落新应用构建方式。
- 参考回答：
  - 我第一批亲手落地的动作比较集中。先是把 Monorepo 的仓库结构搭起来，把应用层和共享包层分开；接着规划 `apps` 和 `packages` 的边界，避免一上来就乱堆；然后优先拆第一批低风险公共包，也就是 `types` 和 `util`，等这部分稳定之后再逐步往 `api`、`ui` 和业务包扩展；另外我还把 `turbo` pipeline 配起来，让根命令可以统一触发任务，新应用的构建方式也是在这阶段落进去的。整体上我先解决的是骨架问题，而不是一开始就追求所有包都拆完。

## 风险质疑题

### 9. 你是不是把 Monorepo 写得太大了？到底哪些成果你不该说？

- 分类：风险质疑
- 风险等级：高
- 回答锚点：`gel-workspace-monorepo-facts.md` -> `暂不写入简历的内容`
- 答题提示：
  - 不说 `45 分钟到 5 分钟`、`CI 95%+`、`磁盘占用降低 60%`。
  - 不说 `架构师`、`项目经理`。
  - 不说自己独立完成了全部迁移。
- 参考回答：
  - 这个项目我确实会主动收边界，避免写太大。最典型的就是那些未经重新确认的量化结果，比如 `45 分钟到 5 分钟`、`CI 95%+`、磁盘占用下降多少，这些我现在都不会直接写。角色上我也不会写成“架构师”或者“项目经理”，更不会说自己独立完成了全部迁移。更真实的说法是，我主导了迁移方案，并完成了仓库结构、拆包顺序、`turbo` pipeline 和新应用构建这些关键落地。这样写虽然克制一些，但证据链更完整，面试里更站得住。

### 10. 为什么没有顺手把旧主应用一起切到 Vite？

- 分类：风险质疑
- 风险等级：中
- 回答锚点：`gel-workspace-monorepo-facts.md` -> `关键决策 / 5. 不强推旧主应用立即切到 Vite`
- 答题提示：
  - 影响面过大，需要详细测试和改造。
  - 当时优先级更高的是统一工作区和复用路径。
  - 这是明确取舍，不是遗漏。
- 参考回答：
  - 我没有顺手把旧主应用一起切到 Vite，不是没想到，而是我明确把它压后了。因为旧主应用本身历史更重，直接切构建体系影响面很大，测试和兼容成本都不低。如果当时把这个事情和 Monorepo 迁移绑在一起做，项目风险会明显抬高。所以我的取舍是先解决统一工作区、公共能力复用和任务编排这些更直接的问题，让新项目先跑起来，旧主应用的构建升级后续再看时机单独处理。这个决定本质上是控制迁移边界，不是能力缺失。
