# Report Print 事实校准稿

> 用途：作为简历与面试表述的事实底稿。
> 原则：只记录已确认的真实情况；如果与包装稿冲突，以本稿为准。

## 项目背景

- 这是一个已有的遗留打印项目，不是从 0 到 1 新建。
- 项目目标是为企业征信报告提供 PDF 打印与预览能力。
- 我接手时，项目整体偏静态，缺少打包和工程化支持，维护成本较高。
- 当时常见工作内容包括补齐数据、对准样式，以及处理复杂打印场景。

## 我的角色边界

- 我是这个项目实现侧的 owner。
- 我主导了项目中的核心渲染逻辑设计与实现。
- 我主导了三层分页架构的设计与实现。
- 我主导了 HTML 感知的行分割方案设计与实现。
- 我的主要工作重心在开发实现和可维护性改造。
- 需求理解、业务定义类工作不是我的主要贡献点，不应过度写到我名下。

## 关键决策

### 1. 先恢复可维护性，而不是直接在旧结构上继续堆功能

- 我接手时，旧项目没有清晰的逻辑分层。
- 数据加载链路较混乱。
- 节点渲染没有做好隔离和抽象，存在直接按 DOM 节点挂载内容的情况。
- 核心逻辑集中在一个约 2 万行的大文件中，嵌套深、跳转多、结构不清晰。
- 在这种状态下直接继续开发新功能，问题定位和后续演进成本都很高。

### 2. 代码拆分先按逻辑职责，再按渲染节点继续细化

- 第一层拆分不是按文件大小机械切分，而是按逻辑职责拆开。
- 逻辑职责这一层稳定后，再按具体渲染节点和模块继续细化。
- 一些没有业务入口的旧逻辑直接删除。
- 一些明显重复且质量更差的旧实现也直接删除。

### 3. 分页不能继续依赖纯文本长度估算

- 旧思路按纯文本长度估算切分点，会严重偏离真实渲染结果。
- 很多单元格内容本身带外层 HTML 结构，粗暴按纯文本切分会破坏标签完整性。
- 这类错误会直接导致跨页后的渲染问题。

### 4. 采用三层分页架构处理长表格跨页

- 我主导决定并实现了三层分页架构。
- 这一层架构把页面管理、表格处理、行内分割拆开，避免把所有逻辑继续堆在一起。
- 目标是让常规分页和极端行处理各自落在更清晰的职责边界内。

### 5. 采用 HTML 感知的行分割方案处理极端行

- 我主导设计并实现了 `splitForSingleLineFit` 这类 HTML 感知分割方案。
- 核心做法不是按纯文本处理，而是先把单元格内容拆成可操作的 HTML 单元。
- 然后通过迭代试探适配和溢出检测，得到当前页可容纳部分和剩余部分，交给后续分页流程继续处理。

## 落地动作

1. 逐步把旧项目中的巨型文件按大模块拆开。
2. 梳理原有渲染流程，删除无业务入口和明显重复的旧逻辑。
3. 重建分页相关的职责边界，形成页面管理、表格处理、行分割的分层结构。
4. 设计并实现 HTML 感知的行分割方案，替代纯文本长度估算。
5. 将行分割结果接入分页流程，使极端行可以被拆成当前页内容和剩余内容继续处理。

## 拆解判断标准

- 遇到遗留逻辑时，优先判断它是否仍有业务入口。
- 没有业务入口的逻辑可以直接删除。
- 对重复实现，优先保留更清晰、更稳定的一版，删除明显更差的旧实现。
- 在分页问题上，优先保证渲染结构正确，再考虑局部实现细节。

## 取舍与非目标

- 这个项目不是以需求定义或业务抽象为主的项目，不应包装成“业务 owner”。
- 我不应把所有结果包装成大规模量化收益。
- 我在这个项目中最核心的价值是实现侧接管、可维护性恢复，以及分页与行分割能力建设。

## 可承担结果

- 解决了富文本和复杂单元格跨页时容易出现的 HTML 结构破坏问题。
- 让分页不再继续依赖纯文本长度估算。
- 让长表格跨页和极端行处理这类逻辑有了更清晰的职责分层。
- 提升了遗留打印项目的可维护性，使后续继续补数据和对样式更可控。
