# AI Chat 面试问题

> 基于 [ai-chat-facts.md](./ai-chat-facts.md) 的第一版项目问答稿。

## 基础题

### 1. 这个 AI Chat 项目你最核心做了什么？

- 分类：基础
- 风险等级：低
- 回答锚点：`ai-chat-facts.md` -> `我的角色边界`
- 答题提示：
  - 主导 AI Chat 页面整体方案和关键实现。
  - 主导 SSE 流程和对话状态收敛。
  - 主导消息模型和结构化渲染抽象。
- 参考回答：
  - 这个项目我最核心做的，不是单独做一个聊天输入框，而是把整个 AI 对话页面真正搭起来。具体包括三块：第一是把发送、流式返回、中断、恢复这些会话流程组织起来；第二是把页面里真正需要保存的状态收敛住，避免 UI 状态越做越散；第三是把消息模型和结构化渲染抽出来，让它不只是能展示纯文本，而是能承载更复杂的消息类型。整体上我更像这个页面实现侧的 owner。

### 2. 这个项目为什么不只是一个聊天页面？

- 分类：基础
- 风险等级：中
- 回答锚点：`ai-chat-facts.md` -> `项目背景`
- 答题提示：
  - 不只是输入输出，还要把流式响应、对话状态、结构化消息和页面交互整合成一个整体。
  - 既有页面体验复杂度，也有消息模型和渲染复杂度。
- 参考回答：
  - 因为这个项目真正难的地方，不是把输入框和消息列表摆出来，而是要把很多东西收成一个完整系统。用户发起一次对话之后，后面会有流式返回、会话状态切换、中断恢复、消息落定这些流程；与此同时，页面上承载的也不只是纯文本，还会有结构化消息和更复杂的渲染。所以它不是普通聊天壳子，而是一个把交互流程、状态组织、消息抽象和页面渲染一起落地的项目。我的工作重点也正是在把这些环节组织成可工作的整体。

## 决策题

### 3. 为什么你最先做的是逻辑分层，而不是直接继续堆状态？

- 分类：决策
- 风险等级：高
- 回答锚点：`ai-chat-facts.md` -> `关键决策 / 1. 先做逻辑分层`
- 答题提示：
  - 流式消息生命周期长，UI 状态容易发散。
  - 中断、恢复、持续流式更新的状态切换复杂。
  - 能推导出的内容就不重复存储，先收敛真正需要存的状态。
- 参考回答：
  - 我当时最先做逻辑分层，是因为这个项目最危险的不是功能点不够，而是状态很容易越堆越乱。流式消息从发送到结束，里面会经过处理中、流式中、取消、失败、完成这些状态，如果页面上再混着各种局部 UI 状态，很快就会互相打架。所以我先把状态边界收紧，只保留真正要持久化和驱动流程的状态，能从现有数据推导出来的就不重复存。这样后面无论是中断恢复还是结构化消息接入，都会稳很多。

### 4. 你这里的分层具体是什么？

- 分类：决策
- 风险等级：中
- 回答锚点：`ai-chat-facts.md` -> `关键决策 / 2. 聊天能力按分层架构组织`；`private/carrier/assets/gel-workspace/packages/gel-ui/docs/biz/ai-chat/design/architecture-design.md`
- 答题提示：
  - `UI / 状态 / 逻辑 / 服务` 四层。
  - 页面展示、状态管理、流程编排、服务处理分层组织。
  - 目标是避免流式交互和页面状态全部耦合在一起。
- 参考回答：
  - 我这里大致拆成四层。最上面是 UI 层，负责消息展示、输入区和交互容器；再下面是状态层，主要管会话状态、发送状态这些页面要消费的数据；第三层是逻辑层，负责把一次对话从开始到结束串起来，处理事件和流程编排；最下面是服务层，负责预处理、预检、流式执行和结果落库这类能力。这样拆的好处是页面不会直接跟底层流式细节耦在一起，问题排查和后续扩展都更清楚。

### 5. 为什么对话流程要按流水线组织？

- 分类：决策
- 风险等级：中
- 回答锚点：`ai-chat-facts.md` -> `关键决策 / 3. 对话流程按流水线组织`；`private/carrier/assets/gel-workspace/packages/gel-ui/docs/biz/ai-chat/design/process-flow.md`
- 答题提示：
  - 预处理、预检、流式准备/执行、结果处理分阶段。
  - 让消息进入系统后的状态变化可追踪。
  - 避免状态散落在页面逻辑中。
- 参考回答：
  - 我把会话流程组织成流水线，核心是想让一次消息从进入系统到结束，整个生命周期都可追踪。大致会先做预处理，比如输入校验和标准化；然后做预检，处理权限和内容检查；通过以后进入流式准备，把上下文和请求参数拼好；接着进入真正的 SSE 流式执行，持续接收 chunk；最后在结果处理阶段更新消息状态、落最终结果。这样每个阶段职责都比较单一，出问题时也能很快判断是前置校验、流式阶段还是结果收尾出了问题。

### 6. 为什么消息模型和结构化渲染要单独抽象？

- 分类：决策
- 风险等级：高
- 回答锚点：`ai-chat-facts.md` -> `关键决策 / 4. 消息模型和结构化渲染需要统一抽象`；`private/carrier/assets/gel-workspace/packages/gel-ui/docs/biz/ai-chat/design/message-model-design.md`
- 答题提示：
  - 不是纯文本聊天，还要承载结构化消息。
  - 不能每种消息类型都写一套页面逻辑。
  - 需要统一进入可解析、可渲染、可扩展的体系。
- 参考回答：
  - 因为这个项目不是纯文本聊天，如果每来一种消息类型就直接在页面里加一个特殊分支，后面一定会失控。所以我会先把消息统一抽象成基础消息模型，再在这个基础上扩展文本、结构化消息、交互型消息这些类型。进入渲染前也会先走一层解析，把类型识别、校验、转换和降级处理收敛起来。这样页面消费的是统一模型，而不是一堆零散接口字段，后面新增消息类型时也不会把页面逻辑再次打散。

## 深挖题

### 7. 这个项目里你实际做了哪些落地动作？

- 分类：深挖
- 风险等级：中
- 回答锚点：`ai-chat-facts.md` -> `落地动作`
- 答题提示：
  - 搭整体交互流程。
  - 组织 SSE 生命周期和对话状态。
  - 收敛状态。
  - 设计消息模型。
  - 推进结构化消息渲染接入页面。
- 参考回答：
  - 我实际落地的事情主要有几类。第一是把整个页面交互流程搭起来，包括发送、流式返回、结束和异常场景；第二是把 SSE 生命周期和页面状态拆开组织，避免所有逻辑都塞在组件里；第三是收敛真正需要存的状态，把一些可以推导的数据拿掉；第四是把消息模型抽出来，让不同消息都能进入统一解析和渲染流程；最后再把结构化消息逐步接进页面，保证它和原来的文本消息可以共存，而不是互相污染。

### 8. 你如何判断哪些东西应该入状态，哪些应该推导？

- 分类：深挖
- 风险等级：高
- 回答锚点：`ai-chat-facts.md` -> `拆解判断标准`
- 答题提示：
  - 首要问题不是组件数量，而是状态发散。
  - 能从已有数据推导出来的内容不重复入状态。
  - 页面逻辑、消息模型、流式请求和结果处理不能全部混在一起。
- 参考回答：
  - 我判断一个东西要不要入状态，核心看两点。第一，它是不是流程推进必须依赖的源状态，比如当前会话是不是在流式中、消息是不是已经完成，这种一般要保留；第二，它是不是页面随时可以从已有数据算出来，如果能推导出来，我就尽量不单独存。因为 AI Chat 这种场景里，重复状态一多，很容易出现一个地方更新了、另一个地方没跟上，最后 UI 和实际流程不一致。我更倾向让状态少而稳，其他信息在渲染阶段按需派生。

## 风险质疑题

### 9. 你为什么不把这个项目写成“彻底解决页面抖动”？

- 分类：风险质疑
- 风险等级：高
- 回答锚点：`ai-chat-facts.md` -> `取舍与非目标`、`暂不写入简历的内容`
- 答题提示：
  - 滚动优化只是整体流式交互稳定性中的一个局部专题。
  - 你更真实的主轴是整体聊天流程搭建、状态收敛和消息模型设计。
  - 不要用绝对化结论描述局部问题。
- 参考回答：
  - 我不把它写成“彻底解决页面抖动”，主要是因为这会把一个局部问题说得过满。滚动和渲染稳定性当然是项目里的一部分，但它本质上挂在更大的主线上，就是整个流式对话页面怎么稳定运转。如果我直接拿“彻底解决抖动”做主轴，面试官很容易继续追问具体指标和边界，反而把项目说窄了。更真实的表达应该是：我主导了整体聊天流程、状态收敛和消息模型设计，页面稳定性是这条主线下的结果之一，而不是唯一成果。

### 10. 这个项目最稳的结果是什么？

- 分类：风险质疑
- 风险等级：中
- 回答锚点：`ai-chat-facts.md` -> `可承担结果`
- 答题提示：
  - 对话页面和流式交互整体能够稳定工作。
  - 结构化消息可以进入统一解析和渲染体系。
  - 状态切换不再那么容易发散。
- 参考回答：
  - 这个项目我最稳的结果，不会写成某个局部性能指标，而是整体对话页面和流式交互能够稳定工作。更具体一点，就是发送、流式更新、中断恢复、结果落定这些环节可以串成一个完整流程，页面不会因为状态太散而很快失控。另外结构化消息也能进入统一的解析和渲染体系，不再是页面里到处塞特殊分支。这个结果虽然没有夸张数字，但在面试里是我愿意承担、也能解释清楚的。
