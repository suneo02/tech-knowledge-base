# AI Chat 事实校准稿

> 用途：作为简历与面试表述的事实底稿。
> 原则：只记录已确认的真实情况；如果与包装稿冲突，以本稿为准。

## 项目背景

- 这是一个企业级 AI 对话页面，服务于自然语言触发分析和数据展示的场景。
- 核心挑战：流式响应（SSE）生命周期长、状态发散、多种结构化消息（表格、图表、建议项）渲染耦合。

## 我的角色边界

- 我主导了 AI Chat 页面整体方案和关键实现。
- 我主导了基于四层架构的逻辑分层与状态收敛。
- 我设计并实现了五阶段流式处理流水线（Pipeline）。
- 我主导了统一消息模型（Unified Message Model）与结构化渲染抽象。

## 关键决策

### 1. 架构显性化：引入 UI / 状态 / 逻辑 / 服务四层架构
- **UI层**：由 `AIChatApp` 容器主导，通过**插槽系统 (Slot System)** 解决不同业务场景的 UI 定制需求。
- **状态层**：建立**分层状态体系 (Global -> Session -> Message -> UI)**，使用 `useChatBase` 等自定义 Hooks 确保状态单向流转，避免 UI 直接操作原始流数据。
- **逻辑层**：引入 **HookBus (事件总线)**，负责对话流程编排与中间件逻辑，实现业务逻辑与展示层的彻底解耦。
- **服务层**：封装 SSE 生命周期，屏蔽网络层细节，提供标准化的数据获取能力。

### 2. 流程标准化：五阶段流式处理流水线
- 将一次对话生命周期严格划分为：**预处理 (Pre-process)** -> **预检 (Pre-check)** -> **流式准备 (Stream Prep)** -> **流式执行 (Stream Exec)** -> **结果处理 (Result Handling)**。
- 状态机流转：`pending` (待处理) -> `processing` (预处理/预检中) -> `streaming` (流式传输中) -> `completed` / `failed` / `cancelled` (终态)。
- 价值：每个阶段职责单一，使得中断恢复、异常捕获和状态回滚变得可预测。

### 3. 模型抽象化：统一消息模型与解析渲染分离
- **统一模型**：所有消息继承自 `BaseMessage`。AI 消息扩展支持 `reasoning` (推理)、`suggestions` (建议)、`entities` (实体) 和 `references` (溯源)。
- **解析流水线**：采用 **Detect -> Validate -> Transform** 模式。原始流数据经过解析器转换为标准模型，解析失败自动触发 **Fallback (降级)** 渲染为纯文本。
- **动态渲染**：通过 `MessageRendererRegistry` 实现类型与组件的动态映射，支持**增量渲染 (Incremental Rendering)**，解决大段流式内容输出时的卡顿问题。

## 落地动作

1. **搭建四层架构体系**：重构原有混乱的组件逻辑，将其拆分为清晰的 UI、State、Logic、Service。
2. **实现流式处理流水线**：编写核心 Handler，规范化 SSE 的开启、监听（`stream:chunk`）、异常及关闭流程。
3. **设计消息 Factory 与 Registry**：实现消息类型的自动识别与渲染组件的动态加载。
4. **性能与体验优化**：引入**溯源标记 (Citations)** 和**实体链接 (Entity Linking)** 渲染逻辑，提升回答的专业性与可交互性。
5. **稳定性保障**：在流水线中加入重试与断点恢复逻辑，提升弱网环境下的对话成功率。

## 拆解判断标准

- **状态入库原则**：只有“流程推进依赖的源状态”入库（如 `isStreaming`），“派生 UI 状态”尽量通过计算得出。
- **组件拆分原则**：展示组件（Dumb Components）不持有任何流控逻辑。
- **抽象边界**：解析器（Parser）只负责数据结构转换，不涉及任何样式定义。

## 取舍与非目标

- **不追求“绝对不抖动”**：滚动优化只是流水线结果的一部分，主轴是逻辑的健壮性。
- **不做“通用聊天框架”**：目前的抽象是为了支撑“企业分析场景”，不强求兼容所有 IM 场景（如语音、群聊）。

## 可承担结果

- 构建了一套**生产级可用的 AI 对话底层协议与组件库**。
- 支持 4+ 种结构化消息类型，接入新消息类型的研发成本降低 50% 以上（只需实现对应的 Parser 和 Renderer）。
- 解决了 SSE 状态丢失和 UI 数据不一致的顽疾，流程节点可追踪、可回溯。
