# AI Chat 事实校准稿

> 用途：作为简历与面试表述的事实底稿。
> 原则：只记录已确认的真实情况；如果与包装稿冲突，以本稿为准。

## 项目背景

- 这是一个企业级 AI 对话页面，服务于自然语言触发分析和数据展示的场景。
- 项目核心不只是一个聊天输入框，而是要把流式响应、对话状态、结构化消息和页面交互整合成一个可工作的整体。
- 该项目既有页面层面的聊天体验要求，也有消息模型和结构化渲染层面的复杂度。

## 我的角色边界

- 我主导了 AI Chat 页面整体方案和关键实现。
- 我主导了 SSE 流程和对话整体状态的收敛。
- 我主导了消息模型和结构化渲染这套抽象。
- 我可以把这个项目写成“整体流程搭建 + 状态收敛 + 消息模型设计”的 owner 案例。
- 但我不应把某些局部问题包装成绝对化结果。

## 关键决策

### 1. 先做逻辑分层，而不是继续堆状态

- 这个项目的核心难点，是流式消息生命周期很长，UI 状态很容易发散。
- 尤其在中断、恢复、持续流式更新这些场景里，状态切换复杂，如果没有清晰边界，页面交互会变乱。
- 我最先做的不是单纯增加状态，而是先做逻辑分层，尽量收敛真正需要存的状态。
- 能从已有数据推导出来的内容，就尽量不重复存储。

### 2. 聊天能力按分层架构组织

- 这个项目的整体设计更接近 `UI / 状态 / 逻辑 / 服务` 四层。
- 页面展示、状态管理、流程编排、服务处理分别放在不同层次中。
- 这样做的目的，是让流式交互和页面状态不会全部耦合在一个层次里。

### 3. 对话流程按流水线组织

- 我主导的流程设计更接近一条清晰的处理流水线：
  - 预处理
  - 预检
  - 流式准备 / 执行
  - 结果处理
- 这种设计的核心价值，是把消息进入系统后的状态变化组织成可追踪的阶段，而不是散落在页面逻辑中。

### 4. 消息模型和结构化渲染需要统一抽象

- 这个项目不只是纯文本聊天，还需要承载结构化消息。
- 我主导了消息模型和结构化渲染这套抽象。
- 目标是把不同类型的消息统一纳入一套可解析、可渲染、可扩展的体系，而不是让每种消息各写一套页面逻辑。

## 落地动作

1. 搭建 AI Chat 页面整体交互流程。
2. 将对话页面、SSE 生命周期和整体状态组织成更清晰的分层结构。
3. 收敛真正需要持久化的状态，减少可推导数据的重复存储。
4. 设计并实现消息模型，支持不同类型消息的统一表示。
5. 推进结构化消息的解析与渲染接入页面流程。
6. 在流式输出和页面渲染场景中，持续处理状态切换和交互稳定性问题。

## 拆解判断标准

- 这个项目里的首要问题不是组件数量，而是状态发散。
- 能从已有数据推导出来的内容，不应重复入状态。
- 页面逻辑、消息模型、流式请求和结果处理不应全部混在一起。
- 结构化消息场景必须优先保证抽象统一，而不是每次新增类型都写特殊分支。

## 取舍与非目标

- 这个项目不应只被包装成一个“滚动优化”项目。
- 它更适合作为“整体聊天流程搭建 + 流式状态收敛 + 消息模型设计”的案例。
- 我不应直接写“彻底解决页面抖动”这类绝对化结论。
- 我也不应把服务层职责写得过满，比如直接写成“服务层专门负责 SSE 的流式解析和收敛”。

## 可承担结果

- 对话页面和流式交互整体能够稳定工作。
- 结构化消息可以纳入统一的解析和渲染体系。
- 流式消息和 UI 状态之间的关系更可控，状态切换不再那么容易发散。
- 页面渲染和交互稳定性得到改善。
