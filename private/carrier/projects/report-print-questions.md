# Report Print 面试问题

> 基于 [report-print-facts.md](./report-print-facts.md) 的第一版项目问答稿。

## 基础题

### 1. 这个 PDF 打印项目的核心目标是什么？

- 分类：基础
- 风险等级：低
- 回答锚点：`report-print-facts.md` -> `项目背景`
- 答题提示：
  - 企业征信报告的 PDF 打印与预览。
  - 不是从 0 到 1 新建，而是遗留项目接手。
  - 重点是复杂打印场景和可维护性恢复。
- 参考回答：
  - 这个项目的核心目标，是给企业征信报告提供 PDF 打印和预览能力。但我不会把它讲成从零到一新建系统，因为我接手的时候它已经是一个遗留项目了。对我来说，真正的难点有两层：一层是打印场景本身很复杂，尤其是长表格跨页和极端行处理；另一层是旧项目本身可维护性很差，所以我做的事情既包括把复杂分页能力补起来，也包括先把原来那套难以继续开发的结构梳理清楚，不然你后面只会一直在旧坑上补洞。

### 2. 你在这个项目里的角色是什么？

- 分类：基础
- 风险等级：高
- 回答锚点：`report-print-facts.md` -> `我的角色边界`
- 答题提示：
  - 实现侧 owner。
  - 主导核心渲染逻辑、三层分页架构、HTML 感知行分割。
  - 不把需求定义和业务理解过度写到自己名下。
- 参考回答：
  - 我在这个项目里的角色，更准确是实现侧 owner。因为项目本来就存在，所以我不会把需求定义和业务抽象都写成我的功劳，但在实现层面，核心渲染逻辑、三层分页架构，以及后面 HTML 感知的行分割方案，都是我主导设计和实现的。我前期花了不少精力先把遗留代码梳理到能继续维护的状态，然后再把复杂分页这条主线搭起来。所以这个项目我最适合承担的，是接手遗留系统后把实现能力真正做扎实，而不是把自己包装成业务 owner。

## 决策题

### 3. 你为什么接手后先做可维护性改造，而不是直接继续补功能？

- 分类：决策
- 风险等级：中
- 回答锚点：`report-print-facts.md` -> `关键决策 / 1. 先恢复可维护性`
- 答题提示：
  - 旧项目没有清晰逻辑分层。
  - 数据加载混乱，节点渲染缺少隔离和抽象。
  - 核心逻辑集中在一个约 2 万行的大文件里。
- 参考回答：
  - 我接手后先做可维护性改造，是因为当时旧项目已经乱到不适合直接继续堆功能了。它没有清晰分层，数据加载链路比较混乱，节点渲染也没做好隔离，很多内容还是直接找 DOM 节点挂进去的。最核心的问题是大量逻辑堆在一个两万行左右的大文件里，嵌套深、跳转也乱。你在这种基础上继续加功能，短期能交付，但后面定位问题和继续演进都会越来越难。所以我先把结构理顺，这是后面做分页能力的前提，不是额外工作。

### 4. 你为什么没有继续用按纯文本长度估算切分点的旧思路？

- 分类：决策
- 风险等级：高
- 回答锚点：`report-print-facts.md` -> `关键决策 / 3. 分页不能继续依赖纯文本长度估算`
- 答题提示：
  - 纯文本切分会偏离真实渲染结果。
  - 很多单元格内容本身带外层 HTML 结构。
  - 粗暴切分会破坏标签完整性并导致跨页渲染错误。
- 参考回答：
  - 我没有继续沿用按纯文本长度估算切分点的思路，核心原因是它和真实渲染之间偏差太大。尤其在这个项目里，很多单元格内容不是一段普通字符串，而是带 HTML 结构的富文本。如果你只按纯文本长度去切，第一会和真实高度不一致，第二很容易把标签结构直接切坏，跨页以后渲染就会出问题。所以我后面改成 HTML 感知的思路，本质上就是让分割逻辑更接近真实 DOM 结构，而不是拿文本长度去猜页面最终长什么样。

### 5. 为什么要做三层分页架构？

- 分类：决策
- 风险等级：中
- 回答锚点：`report-print-facts.md` -> `关键决策 / 4. 采用三层分页架构`；`private/carrier/assets/gel-workspace/apps/report-print/docs/pdf-pagination-process.md`
- 答题提示：
  - 页面管理、表格处理、行分割职责不同。
  - 不拆层就会继续堆在一起，维护困难。
  - 目标是让常规分页和极端行处理都有清晰边界。
- 参考回答：
  - 我做三层分页架构，核心是把不同粒度的问题拆开。最上层是页面管理，负责新建页面和整体高度判断；中间层是表格处理，负责按行追加、检测是否溢出以及常规分页；最底层是极端行处理，也就是一行本身就超过一页时，怎么继续往下拆。如果这三类职责全堆在一起，代码会很快重新变成一团。所以我把常规分页和极端行分割明确区分开，让大多数正常场景走常规流程，只有真正超页的单行才进入更细粒度的拆分逻辑，这样结构会稳很多。

## 深挖题

### 6. 你实现的 HTML 感知行分割，核心动作是什么？

- 分类：深挖
- 风险等级：高
- 回答锚点：`report-print-facts.md` -> `关键决策 / 5. 采用 HTML 感知的行分割方案`；`private/carrier/assets/gel-workspace/apps/report-print/docs/pdf-pagination-process.md`
- 答题提示：
  - 先把单元格内容拆成可操作的 HTML 单元。
  - 再通过迭代试探适配和溢出检测，得到当前页可容纳部分和剩余部分。
  - 分割结果交回分页流程继续处理。
- 参考回答：
  - 这个方案最核心其实就两步。第一步不是直接切文本，而是先把单元格内容拆成更可操作的 HTML 单元，这样后面处理的时候不会一刀把标签结构砍坏。第二步是做迭代试探，把这些内容一点点放到当前页里，通过真实溢出检测去判断还能不能继续放。最后会得到两部分结果，一部分是当前页能放下的 `firstLineData`，另一部分是剩余内容 `remainingData`。前者留在当前页，后者再当成新的待处理行交回分页流程继续跑。

### 7. 你接手后第一批实际做了哪些事情？

- 分类：深挖
- 风险等级：中
- 回答锚点：`report-print-facts.md` -> `落地动作`
- 答题提示：
  - 拆巨型文件。
  - 梳理渲染流程，删除无入口和重复旧逻辑。
  - 重建分页职责边界。
  - 用 HTML 感知方案替代纯文本估算。
- 参考回答：
  - 我接手后第一批工作其实挺明确的。先把那个巨型文件按大模块逐步拆开，不然你根本没法继续维护；然后去梳理原有渲染流程，把已经没有业务入口的逻辑和明显重复、质量更差的旧实现删掉；结构稍微稳一点之后，再把分页相关职责重新划开，区分页面管理、表格处理和行内分割；最后才是把 HTML 感知的行分割方案接进去，用它替代原来靠纯文本长度估算的做法。这个顺序对我来说很重要，因为它先解决了能不能维护，再解决能不能把复杂场景做好。

## 风险质疑题

### 8. 你是不是把结果写得太满了？哪些成果其实不能说？

- 分类：风险质疑
- 风险等级：高
- 回答锚点：`report-print-facts.md` -> `暂不写入简历的内容`
- 答题提示：
  - 不说 `像素级一致性`。
  - 不说 `1000+` 批量导出稳定性。
  - 不说 `30+ 模板`、`500KB vs 3MB+`。
  - 不把项目包装成从 0 到 1 新建系统。
- 参考回答：
  - 这个项目我现在会明确收住一些容易写满的话。比如我不会直接说“像素级一致性”“1000+ 批量导出稳定性”或者“30+ 模板、体积从 3MB 降到 500KB”这类数字，因为这些要么不是我这次校准后愿意承担的结果，要么没有足够证据支撑。我也不会把它包装成从零到一的新系统。更真实的边界是：这是一个遗留打印项目，我作为实现侧 owner，主导了可维护性恢复、三层分页架构和 HTML 感知行分割，这些才是我真正能解释清楚的成果。

### 9. 这个项目里你最稳的可承担结果是什么？

- 分类：风险质疑
- 风险等级：中
- 回答锚点：`report-print-facts.md` -> `可承担结果`
- 答题提示：
  - 解决富文本和复杂单元格跨页时的 HTML 结构破坏问题。
  - 让分页不再依赖纯文本长度估算。
  - 提升了遗留打印项目的可维护性。
- 参考回答：
  - 这个项目里我最稳的结果，是把富文本和复杂单元格跨页时最容易出现的 HTML 结构破坏问题解决掉了，让分页不再继续依赖纯文本长度估算。再往上讲一点，就是原来那套遗留打印逻辑经过分层以后，可维护性也明显更好了，后面继续补数据、对样式或者处理新分页场景，不再只能在一坨旧逻辑上硬改。我不会把它说成什么特别夸张的大规模业务收益，但作为一个实现侧 owner，这个结果我愿意承担，也能把技术路径讲清楚。
