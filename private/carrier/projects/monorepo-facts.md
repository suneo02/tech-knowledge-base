# Monorepo 与工程化质量管理 事实校准稿

> 用途：作为简历与面试中“工程化治理”与“质量管理”的事实底稿。
> 原则：强调工程确定性、CI/CD 自动化以及遇到问题即学习解决的闭环。

## 项目背景

- 随着 AI Chat、报告、打印等多个应用的并行开发，多仓库管理的维护成本和公共逻辑复用难度极高。
- 项目中存在大量重复的 `util`, `types` 和 `api` 定义，缺乏统一的工程化约束。
- CI/CD 流程原本较分散，且在多项目构建时存在严重的重复计算。

## 关键决策

### 1. 落地 Monorepo 渐进式迁移方案
- **核心逻辑**：不追求一次性大一统，而是按依赖层次剥离。
- **动作**：优先拆分底层的 `types` 和 `util` 包，确保核心依赖的单一来源（Single Source of Truth）。

### 2. 建立“质量防线”与工程化治理体系
- **TSC 强制校验**：在 CI 中引入 `tsc --noEmit`，确保全量类型安全，将潜在 Bug 拦截在编译阶段。
- **Storybook 模块化预览**：在 `apps/ai-chat` 等复杂应用中引入 Storybook，实现 UI 组件的隔离开发与文档化复用。
- **Git Actions 自动化流水线**：主导 CI 工作流设计，涵盖 Linting、Type-check 和 Build 自动化。

### 3. 构建性能优化：Turbo + Remote Cache
- **Turbo Pipeline**：通过 `turbo` 编排 `build`, `test`, `lint` 任务。
- **Remote Cache**：利用 Turbo 的远程缓存机制，显著缩减多人协作时的重复构建时间。

## 落地动作与细节

1. **仓库结构化**：规划 `apps/` 与 `packages/` 边界，实现逻辑物理隔离。
2. **CI 流程优化**：编写 Git Actions 脚本，实现按应用过滤的任务执行。
3. **TSC 与质量控制**：在 Monorepo 根目录下配置统一的类型检查规范。
4. **组件开发标准化**：通过 Storybook 承载业务组件预览（如 Conversation, Message 等）。
5. **构建速度攻坚**：引入 Turbo 远程缓存解决多人构建资源浪费。

## 学习成长与学习闭环

- **遇到问题即深挖**：
  - 遇到构建卡顿 -> 学习并落地 Turbo Remote Cache。
  - 遇到类型定义混乱 -> 深入 Monorepo 物理隔离与 `tsc` 全量校验。
  - 遇到组件复用难 -> 引入 Storybook 建立可视化组件库。
- **结论**：在企业级开发中，我不仅是完成业务，更是在遇到工程瓶颈时，主动学习并引入对应的质量管理工具。

## 可承担结果

- 建立了多应用并行的 Monorepo 工作区，逻辑复用率显著提升。
- CI/CD 流程实现了全量自动化类型安全检查（tsc）。
- 构建耗时在多人协作场景下显著下降。
- 业务组件通过 Storybook 实现了标准化复用。

## 跳槽主线提示

- 简历项目条目优先强调：
  - 工程化视角：Monorepo + Turbo Pipeline
  - 质量确定性：Git Actions CI + TSC 全量校验 + Storybook
  - 学习驱动：遇到构建/复用/协作瓶颈时的工具链快速落地能力。
