# Monorepo 与工程化质量管理 面试问题

> 基于 [monorepo-facts.md](./monorepo-facts.md) 的第一版项目问答稿。

## 基础题

### 1. 你在这个工程化项目里主要做了什么？
- **回答锚点**：`monorepo-facts.md` -> `落地动作与细节`
- **参考回答**：
  - 我主导了从多仓库到 Monorepo 的渐进式迁移方案。这不仅是把物理目录合在一起，更核心的是通过 **Turbo Pipeline** 和 **Git Actions** 建立了一套自动化的质量管理体系。我负责规划了 `apps/` 与 `packages/` 的隔离边界，并优先剥离了 `types` 和 `util` 以实现单一来源。同时，我为了解决构建效率问题，落地了 **Turbo Remote Cache**，并为了组件复用推行了 **Storybook**。

### 2. 你理解的“质量管理”在前端工程中是如何体现的？
- **回答锚点**：`monorepo-facts.md` -> `关键决策 / 2`
- **参考回答**：
  - 我理解的质量管理不是事后修补，而是在开发阶段通过工具链建立“自动化防线”。
  - 比如我推行了 **TSC 全量校验（tsc --noEmit）**，确保类型安全是所有 CI 任务的前置条件；我也主导了 **Git Actions CI/CD**，让 lint、type-check 和 build 自动化运行，不给线上留隐患。此外，引入 **Storybook** 也是质量管理的一部分，它通过隔离开发减少了 UI 组件的不可预见副作用。

## 决策题

### 3. 为什么引入 Turbo Remote Cache？它解决了什么？
- **回答锚点**：`monorepo-facts.md` -> `关键决策 / 3`
- **参考回答**：
  - 在多应用 Monorepo 环境中，一旦项目变多，全量构建非常耗时，且很多构建结果其实在同事或 CI 机器上已经生成过了。我当时发现大家在重复构建相同内容，所以引入了 **Turbo Remote Cache**。它通过云端缓存任务产物，让团队成员可以共享构建结果。遇到构建瓶颈时，我没有被动等待，而是主动去学习并落地了这套分布式缓存方案，显著提升了协作效率。

### 4. 为什么要在 CI 中强制加入 TSC 检查？
- **回答锚点**：`monorepo-facts.md` -> `落地动作与细节 / 3`
- **参考回答**：
  - 在大型多人协作项目中，仅仅靠本地 IDE 的提示是不够的，很容易出现“本地没报错，合入后线上挂掉”的情况。所以我把 **`tsc` 校验** 作为 CI/CD 工作流的硬性门槛，确保所有代码必须在类型层面完全收敛。这是我对“工程确定性”的追求，也是我这两年建立起来的质量保障意识。

## 驱动与学习

### 5. 你是怎么体现“遇到问题就学习解决”的？
- **回答锚点**：`monorepo-facts.md` -> `学习成长与学习闭环`
- **参考回答**：
  - 在企业级应用开发中，我会习惯性地去深挖瓶颈背后的底层原理。比如当多个项目需要复用同一个组件且逻辑冲突时，我没有简单地通过复制黏贴解决，而是去研究了 **Storybook** 的最佳实践和 **Monorepo** 的包边界划分。再比如为了解决 CI 构建慢，我去系统学习了任务流编排。我更倾向于通过工程化的手段彻底解决一类问题，而不是单点修补。
