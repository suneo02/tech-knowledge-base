## 基础题

### 1. 企业详情页这个项目你到底做了什么？

- 分类：基础
- 风险等级：高
- 回答锚点：`company-facts.md` -> `我的角色边界`
- 答题提示：
  - 这是长期存在的企业级详情页，不是从 0 到 1 新建。
  - 你的角色是核心开发维护者。
  - 核心价值是持续交付、处理复杂业务逻辑、做渐进式改造。
- 参考回答：
  - 这个项目我不会包装成从零到一重构，更真实的说法是我是这个企业详情页的核心开发维护者。它本来就长期存在，业务模块多、历史逻辑也很多，我主要是在这个基础上持续做需求交付，同时处理一些复杂分支和遗留问题。我的价值点也不是单独做了某个特别炫的技术专题，而是在一个复杂企业级页面里，把需求稳定落下去，并且逐步把一些边界比较清楚的历史逻辑拆开，让后续维护和改动的影响面更可控。

- 风险等级：中

  - 历史分支多，改一处容易影响全页。

- 参考回答：

## 决策题

### 3. 为什么这个项目不适合做成一次性重构？

- 分类：决策
- 风险等级：中
- 回答锚点：`company-facts.md` -> `关键决策 / 1. 不把项目写成一次性重构`
- 答题提示：
  - 页面一直存在，历史逻辑分支很多。
  - 一次性推倒重来风险高，影响面大。
  - 更现实的策略是在已有框架内逐步收敛问题。
- 参考回答：
  - 这个项目不适合一次性重构，原因很现实。它不是一个边界特别干净的新页面，而是已经长期在线、承载很多业务分支的企业详情页。你如果在这种项目里直接推倒重来，先不说成本，光是影响面和回归风险就很大，而且很多历史逻辑没那么容易一次说明白。所以我更倾向在现有框架里渐进式处理：一边继续交付业务，一边把比较清楚、比较独立的问题逐步拆掉。这样节奏更稳，也更符合企业项目里真正能落地的做法。

### 4. 长页面问题为什么不直接用虚拟列表？

- 分类：决策
- 风险等级：中
- 回答锚点：`company-facts.md` -> `关键决策 / 3. 长页面优化优先采用占位和延迟加载`
- 答题提示：
  - 需求不是单纯减少 DOM 数量。
  - 用户往下滚动时要立刻看到模块占位和表头。
  - 页面结构需要保持连续。
- 参考回答：
  - 我没有直接用传统虚拟列表，核心不是我不知道这个方案，而是它和这个页面的真实需求不完全匹配。这个页面要求滚动条高度准确且支持快速定位模块占位，虚拟列表会破坏这种结构连续性。
  - 我落地的是基于 `IntersectionObserver` 的方案。相比传统的同步 `scroll` 监听，它最大的优势是异步且非阻塞。旧逻辑在滚动时频繁触发 `getBoundingClientRect` 会导致严重的强制重排（Layout Thrashing），主线程被阻塞。换成原生交叉检测 API 后，配合 `rootMargin` 预加载，既消除了布局抖动，又保证了用户向下滚动时能立刻看到骨架屏占位，平衡了首屏压力与滚动流畅度。

### 5. 你为什么没有先设计一套完整的新抽象体系？

- 分类：决策
- 风险等级：高
- 回答锚点：`company-facts.md` -> `关键决策 / 4. 历史分支逻辑的处理方式是渐进式拆分`
- 答题提示：
  - 核心痛点是揉杂了大量业务逻辑的“上帝组件”。
  - 策略是“冻结旧逻辑，抽取真共性，隔离新分支”。
  - 目标是物理阻断膨胀，而不是一次性翻新。
- 参考回答：
  - 我没有一上来就做整套抽象，是因为那个页面里有一个积累了很久的“上帝组件”，里面沉淀了太多无法舍弃的兜底逻辑。如果我追求一次性重写，回归成本和线上风险根本不可控。
  - 我的做法是“逻辑旁路”：先把真正通用的能力（比如权限、懒加载）从上帝组件里下沉成 Hooks；然后对于新模块，我坚决不再往旧代码里塞 `if-else`，而是开辟独立的渲染分支，直接消费这些下沉的基础能力。这种做法虽然不炫，但它实实在在地阻断了历史逻辑的无限膨胀，让后续需求的影响面变得极其清晰。

## 深挖题

### 6. 这个项目里你实际做的渐进式改造是什么？

- 分类：深挖
- 风险等级：中
- 回答锚点：`company-facts.md` -> `落地动作`
- 答题提示：
  - 针对性能：落地基于 `IntersectionObserver` 的异步延迟加载，消除布局抖动。
  - 针对维护性：对上帝组件做“能力下沉”与“逻辑旁路”重构。
- 参考回答：
  - 我做的改造核心是两个动作。第一是在性能侧，用 `IntersectionObserver` 替换了原本高危的同步 `scroll` 方案，消除了长页面的滚动掉帧；第二是在维护侧，针对消费层的“上帝组件”做了逻辑剥离。我通过提取公共 Hooks 来承载懒加载和权限等切面逻辑，并为新需求建立隔离的渲染路径。这样做让旧代码“原地封存”，新业务“旁路运行”，真正做到了在不推翻系统的前提下恢复了可维护性。

### 7. 你拆分历史分支逻辑时，判断标准是什么？

- 分类：深挖
- 风险等级：高
- 回答锚点：`company-facts.md` -> `拆解判断标准`
- 答题提示：
  - 区分“真正的通用逻辑”与“特异的业务表现”。
  - 优先下沉高频、纯度高的通用能力（权限、性能机制）。
  - 目标是让新需求的影响面不再波及旧组件。
- 参考回答：
  - 我的标准其实就是“解耦通用能力，隔离业务特异”。我会优先去看哪些逻辑是页面层级真正共享的，比如权限校验、懒加载机制、默认列展示逻辑，这些我会毫无保留地抽离出来做底层基建。而对于那些特定业务模块的特殊交互，我不会试图去统一它们，而是用“旁路渲染”的方式让它们独立运行。这样即便某个业务模块逻辑再复杂，它也不会污染到其他模块。我的最终目标是让后续每个新需求的上线，对老代码的回归压力降到最低。
