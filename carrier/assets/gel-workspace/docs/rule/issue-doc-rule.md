# Issue 文档编写规范

## 快速准则
- 同一问题的背景、分析、方案、验证写在一份 Markdown，控制在 1 页（≤160 行）。
- 用表格和要点描述现象、根因、影响范围，所有结论必须给出来源（文档、代码、日志）。
- 内容聚焦功能缺陷与可观察行为，避免记录性能/监控/运维等非功能信息。
- 文档与代码互相链接：正文指向 `路径:行号`，代码中的 TODO/FIXED 指回本 Issue。

## 推荐结构
1. **问题概览**（表格）：标题、状态、优先级、责任人、发现时间、目标上线时间。
2. **背景与预期**：2-3 句说明业务场景与设计基线。
3. **问题陈述**：分别描述现象、根因、影响；根因附上代码或日志路径。
4. **参考资料**：表格列出相关文档/代码、作用、备注。
5. **解决方案**：最终方案要点、负责人、计划时间；如有备选方案，说明放弃理由。
6. **验证记录**：≤5 条验证步骤，覆盖修复的核心流程；如仍有非功能风险，单行备注。
7. **更新日志**：日期 + 事件 + 描述。
8. **附录（可选）**：必要截图或 ≤10 行代码片段。

## 工作流
- 文件命名：`[模块]-issues.md`；进行中放 `apps/<app>/docs/issues/`，归档后移至 `archived/`。
- 状态标识：🚧 进行中 / ✅ 已解决 / ✅ 已归档；优先级用 🔴/🟡/🟢。
- 解决后补充验证记录（只记录功能点），更新代码或文档链接，并在 PR 描述中引用 Issue。

## 检查清单
- [ ] 现象、根因、影响均已列出且带来源。
- [ ] 方案可执行，负责人与时间清晰。
- [ ] 验证步骤覆盖主要功能路径，必要的剩余风险单行说明。
- [ ] 文档长度、格式符合要求，未出现长段落。
- [ ] 代码与文档建立双向引用。
