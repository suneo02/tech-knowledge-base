# 通用基线 (The Baseline)

_适用于所有文档格式的核心原则与语言规范。_

## 1. 核心原则 (Core Principles)

- **真实溯源**：拒绝凭空捏造。所有数据需基于**真实源码**或**可推导逻辑**（如本地 Lighthouse 跑分、DevTools 录制）。
- **STAR 结构**：
  - **S (Situation)**：背景/量级（用户量、数据规模、业务场景）。
  - **T (Task)**：职责（具体模块、功能）。
  - **A (Action)**：技术方案（React/Vue、构建工具、优化策略）。
  - **R (Result)**：量化成果（首屏时间 ms、包体积 %、转化率 %）。
- **技术聚焦**：聚焦主流生态，体现工程化（构建/测试/状态管理）与性能优化能力。

## 2. 语言避坑 (Language Rules)

| ❌ 避免 (模糊/被动)              | ✅ 推荐 (具体/主动)               | 修正示例                                    |
| :------------------------------- | :-------------------------------- | :------------------------------------------ |
| **名词**：系统、平台、机制、体系 | **技术词**：Redis、Webpack、CI/CD | "智能推荐系统" → "基于协同过滤的商品推荐"   |
| **动词**：负责、参与、了解、协助 | **动作**：开发、重构、封装、接入  | "负责用户模块" → "开发登录/注册等 5 个页面" |
| **形容**：高效、深度、完善、优秀 | **数据**：300ms、95%、50%、<2s    | "深度优化性能" → "首屏时间从 3s 降至 1s"    |

## 3. 经验分层指引 (Experience Level Focus)

不同年限在编写各类文档时，侧重点应有所不同：

- **1-3 年**：侧重 **Task & Action**（独立模块、主流框架熟练度、代码规范）。
- **3-5 年**：侧重 **Action & Result**（性能优化、技术攻坚、工程化搭建、组件库沉淀）。
- **5+ 年**：侧重 **Situation & Whitepaper**（技术选型、架构决策、基础设施、团队影响力）。
