# 格式二：项目深挖底稿 (The Manuscript)

- **定位**：面试口述脚本 & 答辩思维导图。
- **目标读者**：面试官 (深度追问)、自己 (临场记忆)。
- **前置要求**：本底稿基于 [通用基线 (The Baseline Protocol)](./project-experience-baseline.md) 的 **STAR-Plus 标准** 进行口语化扩充。
- **AI 生成指令**：作为“编剧”与“辩论教练”，将冷冰冰的技术点转化为有血有肉的“英雄之旅”故事，并构建防御工事。

## 1. 叙事构建逻辑 (Narrative Construction)

### 模块一：电梯演讲 (The Hook / 1 min)

**目标**：在 60 秒内建立“高 P”人设，引发面试官兴趣。

- **生成公式**：`[业务量级背景] + [我的核心职责] + [最大痛点] + [我的解决方案] + [最终收益]`
- **口语化技巧**：使用连接词（“当时遇到的最大问题是...”、“为了解决这个...”、“结果非常有意思...”）。

### 模块二：STAR 深度复盘 (The Deep Dive)

**目标**：对基线中的 STAR-Plus 结构进行**细节填充**，证明亲历。

- **S (背景 - 增强版)**：
  - 不要只说量级，要描述**场景痛点**。（e.g., "在低端机上 FPS 只有 20，用户根本没法滑"）。
- **T (任务 - 增强版)**：
  - 强调**限制条件**。（e.g., “不仅要流畅，还要兼容 IE11”）。
- **A (行动 - 核心)**：
  - **拒绝流水账**：不要简单列举基线中的技术词。
  - **强调决策**：为什么选 A 不选 B？（e.g., “我本来想用虚拟滚动，但发现会导致搜索功能失效，所以改用了分页加载...”）
  - **强调困难**：描述过程中的“坑”和“回滚”。
- **R (结果 - 增强版)**：
  - 除了数据，加上**用户反馈**或**团队评价**。
- **L (资产 - 增强版)**：
  - 提到你留下的文档或规范对团队长期的影响。

## 2. 防御性推演 (Defensive Strategy)

**AI 指令**：分析项目的薄弱环节，生成“刁钻”的追问与“高情商”的回击。

### 策略 A：技术选型质疑 (Trade-off)

- **Trigger**：项目中使用了非主流技术或显得“过度设计”。
- **Q (预设)**：为什么这点数据量要用 Redux？Context 不够吗？
- **A (脚本)**：承认成本 -> 抛出场景 -> 对比收益。
  - "确实，对于小项目 Context 够了。但我们的业务涉及到**多层级组件频繁更新**，Context 会导致全量 Re-render 性能问题（Benchmark 显示慢了 200ms）。Redux 的 selector 机制能精准控制更新..."

### 策略 B：深度原理挖掘 (Under the Hood)

- **Trigger**：简历中提到了具体库（如 React-Virtual）。
- **Q (预设)**：虚拟列表快速滚动白屏怎么解决？原理是什么？
- **A (脚本)**：
  - _原理_：解释 transform 偏移量计算。
  - _实战_：提到 `overscan` (缓冲区) 的配置调优过程。

### 策略 C：不足与反思 (Deficiency & Pivot)

- **Trigger**：项目完美得不真实。
- **Q (预设)**：这个项目有什么遗憾吗？
- **A (脚本)**：
  - **真诚的不足**：测试覆盖率低 / 移动端适配细节不够 / 构建速度后期变慢。
  - **Pivot (改进)**：描述后续的学习计划或补救措施（“所以我最近在看 Vitest...”）。

## 3. 模拟对练 (Simulation)

请 AI 扮演**压力面试官**，针对生成的底稿进行 3 轮追问：

1.  **Round 1 (基础)**：考察 API 熟练度。
2.  **Round 2 (场景)**：修改业务条件（“如果数据量大 10 倍怎么办？”）。
3.  **Round 3 (底层)**：考察源码理解（“Vue 的 diff 算法在这里是怎么运作的？”）。
