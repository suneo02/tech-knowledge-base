# 前端简历项目经历规范

## 核心原则

### STAR 结构

- **Situation**：项目背景（用户量、业务场景、数据规模）
- **Task**：你的职责（负责的具体模块和功能）
- **Action**：技术方案和实现细节
- **Result**：量化成果（性能数据 + 业务指标）

### 量化指标

- 性能：首屏时间、FCP、LCP、包体积、响应时间
- 业务：转化率、留存率、PV/UV、处理量
- 效率：开发时间、代码复用率、构建时间

### 技术栈

- 聚焦主流框架生态（React/Vue + 工具链）
- 体现工程化能力（构建、状态管理、测试）
- 展示性能优化（SSR、懒加载、缓存）

## 语言规范

### 避免使用的词语

**假大空词汇**（尽量避免或替换为具体描述）：

- ❌ 系统、平台、体系、架构（除非确实是架构级工作）
- ❌ 智能、自动、高效、先进
- ❌ 机制、策略、方案（改用具体做法）
- ❌ 深度、全面、完善、优秀
- ❌ 赋能、驱动、助力、打造

**推荐替换方式**：

- "智能推荐系统" → "基于协同过滤的商品推荐"
- "自动化部署机制" → "通过 GitHub Actions 实现 CI/CD"
- "高效缓存策略" → "使用 Redis 缓存热点数据，命中率 85%"
- "完善的错误处理" → "捕获并上报错误，覆盖 95%异常场景"
- "深度优化性能" → "通过虚拟滚动将渲染时间从 2s 降至 200ms"

### 推荐使用的表达

**具体动词**：

- 开发、实现、构建、搭建、封装
- 优化、重构、迁移、升级
- 设计、编写、集成、配置
- 减少、提升、降低、缩短

**具体技术细节**：

- 说明用了什么技术：React Hooks、Webpack、Redis
- 说明做了什么事：拆分组件、配置路由、写接口
- 说明解决了什么问题：内存泄漏、页面卡顿、接口超时
- 说明达到了什么效果：加载时间从 3s 降至 1s

## 标准格式

```
【项目名称】 | 时间跨度
项目简介：项目类型 + 用户规模 + 核心价值
技术栈：框架 + 状态管理 + 构建工具 + UI 库
职责与成果：
- 开发 XX 模块，实现 XX 功能，首屏时间 X ms，转化率提升 X%
- 优化 XX 性能，通过 XX 方案将 XX 从 X 降至 Y
- 搭建 XX 体系（组件库/监控），提升效率 X%
```

## 示例对比

### ❌ 错误示例 1：假大空

```
负责智能推荐系统的架构设计与实现，通过先进的算法和完善的机制，
打造了高效的推荐体系，深度赋能业务增长，全面提升了用户体验。
```

问题：全是空话，看不出做了什么、用了什么技术、达到了什么效果

### ✅ 正确示例 1：具体务实

```
开发商品推荐功能，基于用户浏览历史和协同过滤算法计算推荐结果，
通过 Redis 缓存热点数据（命中率 85%），推荐接口响应时间<200ms，
点击率从 3.2% 提升至 5.8%
```

---

### ❌ 错误示例 2：动词模糊

```
参与电商平台开发，使用 Vue、React 等框架，负责一些页面开发，
学习了很多新技术，提升了自己的能力。
```

问题：动词模糊（参与、负责）、技术堆砌、无量化、关注自我成长

### ✅ 正确示例 2：动词具体

```
【电商平台前端重构】 | 2023.06 - 2023.12
项目背景：100 万+ DAU 的 B2C 电商平台，日均订单 5 万+
技术栈：Vue 3 + TypeScript + Vite + Pinia + Element Plus
职责与成果：
- 重构商品详情页，使用虚拟滚动渲染长列表，首屏时间从 3.2s 降至 1.1s
- 实现图片懒加载和 WebP 格式适配，图片体积减少 45%，LCP 提升 40%
- 封装 15 个业务组件（商品卡片、筛选器、购物车等），新页面开发时间缩短 50%
```

---

### ❌ 错误示例 3：技术词汇滥用

```
构建了完善的前端工程化体系，搭建了智能化的自动部署系统，
建立了高效的性能监控机制，打造了先进的错误追踪平台。
```

问题：每句话都是"系统"、"机制"、"平台"，看不出具体做了什么

### ✅ 正确示例 3：说清楚做了什么

```
配置 Webpack 构建流程，通过代码分割和 Tree Shaking 将包体积减少 30%；
使用 GitHub Actions 实现自动化部署，提交后 5 分钟内完成测试和上线；
接入 Sentry 错误监控，捕获并上报前端异常，覆盖 95% 的错误场景；
集成 Lighthouse CI，每次构建自动检测性能指标，LCP 保持在 2.5s 以内
```

## 动词清单

**推荐使用**（具体、主动）：

- 开发、实现、编写、构建、搭建
- 封装、抽取、拆分、合并
- 优化、重构、迁移、升级
- 配置、集成、接入、部署
- 设计、制定、规范
- 减少、提升、降低、缩短（用于量化成果）

**避免使用**（模糊、被动）：

- 负责、参与、协助、配合
- 了解、学习、接触、熟悉
- 涉及、涵盖、包括
- 助力、赋能、驱动、打造

**对比示例**：

- ❌ "负责用户模块开发" → ✅ "开发用户注册、登录、个人中心等 5 个页面"
- ❌ "参与性能优化工作" → ✅ "通过懒加载和代码分割将首屏时间从 3s 降至 1s"
- ❌ "协助搭建构建系统" → ✅ "配置 Webpack 和 Babel，实现 ES6+ 代码编译"

### 3-5 年

### 5 年以上

- 技术选型和架构决策
- 团队规范和基础设施
- 跨团队协作和影响力

## 常见问题

### Q1：如何避免写成"系统"、"平台"？

不要写：构建了完善的监控系统
应该写：接入 Sentry 错误监控，捕获并上报前端异常

不要写：搭建了自动化部署平台
应该写：使用 GitHub Actions 配置 CI/CD，提交后 5 分钟内自动部署

### Q2：如何把"智能"、"自动"写具体？

不要写：实现智能推荐功能
应该写：基于协同过滤算法实现商品推荐，点击率提升 80%

不要写：自动化测试机制
应该写：编写 Jest 单元测试和 Cypress E2E 测试，覆盖率达 85%

### Q3：如何量化"优化"、"提升"？

不要写：优化了页面性能
应该写：通过虚拟滚动和懒加载，首屏时间从 3s 降至 1s

不要写：提升了开发效率
应该写：封装 20 个通用组件，新页面开发时间从 2 天缩短至 0.5 天

### Q4：预备文档可以详细一些吗？

可以！预备文档（如 project-experience.md）可以保留更多技术细节、架构设计、设计文档引用等内容，方便后续提炼成简历版本。但语言风格仍应务实、具体，避免假大空的词汇。
